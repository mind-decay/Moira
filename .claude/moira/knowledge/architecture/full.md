# Architecture Deep Scan — 2026-03-22 (Previous)

## 1. Service Boundaries

Moira is a three-layer meta-orchestration framework with no runtime server process. It operates entirely within Claude Code sessions via slash commands, agent dispatch, and file-based state.

### Layer 1: Global Layer (`~/.claude/moira/`)
**Responsibility:** System-wide definitions installed once per machine. Read-only at runtime.
**Contents:** Core rules, pipeline definitions, agent role YAML files, skill markdown files, shell libraries, hooks, templates, schemas.
**Source:** `src/` directory in the repository. Installed by `src/install.sh`.
**Interface:** Files read by the orchestrator skill and dispatch module. Shell libraries sourced by hooks and other shell scripts.

### Layer 2: Project Layer (`.claude/moira/`)
**Responsibility:** Per-project configuration, knowledge base, and runtime state. Read-write at runtime.
**Contents:** `config.yaml`, project rules (`project/rules/`), knowledge base (`knowledge/`), MCP registry (`config/mcp-registry.yaml`), task state (`state/`).
**Generated by:** `/moira:init` command (`src/commands/moira/init.md`), which dispatches 4 parallel Explorer agents for bootstrapping.
**Interface:** Read/written by the orchestrator via YAML file manipulation (Read + Write tools, not Bash).

### Layer 3: Execution Layer (Agents)
**Responsibility:** All actual project interaction. Each agent is a Claude Code subagent dispatched via the Agent tool.
**11 agents defined in:** `src/global/core/rules/roles/*.yaml` (all 11 role files present: apollo, hermes, athena, metis, daedalus, hephaestus, themis, aletheia, mnemosyne, argus, calliope).
**Interface:** Agents receive assembled instruction prompts. They write detailed output to `state/tasks/{id}/` files and return a structured status summary to the orchestrator. All role files now include a `response_format` field normalized against `response-contract.yaml` (F-010).

### External Data Source: Ariadne (`.ariadne/`)
**Responsibility:** Project dependency graph. Owned by an external Rust CLI tool (separate repository per D-104).
**Interface:** Moira reads `.ariadne/graph/graph.json` and `.ariadne/views/` but never writes. Orchestrator checks file existence only (not content). Agents receive graph data via assembled instructions.
**Evidence:** `src/global/hooks/guard.sh` lines 67-69 whitelist `.ariadne/` for Read but not Write/Edit.

## 4. Data Flow Paths

### 4a. Task Lifecycle (Standard Pipeline)
```
User input
  -> task.md creates: state/tasks/{id}/input.md, status.yaml, manifest.yaml, current.yaml
  -> Apollo reads input.md, writes classification.md, returns status summary
  -> [GATE: classification]
  -> Hermes reads classification.md, writes exploration.md (parallel)
  -> Athena reads classification.md+input.md, writes requirements.md (parallel)
  -> Metis reads exploration.md+requirements.md, writes architecture.md
  -> [GATE: architecture]
  -> Daedalus reads architecture.md+exploration.md+requirements.md, writes plan.md + instructions/*.md
  -> [GATE: plan]
  -> Hephaestus reads instructions/{agent_name}.md (pre-assembled), writes code to project
  -> Themis reads implementation.md+plan.md+architecture.md, writes review.md + findings/
  -> Aletheia reads implementation.md+review.md, writes test-results.md
  -> [GATE: final]
  -> Mnemosyne reads all state files (background), writes to knowledge/
```

### 4b. State File Flow
- `current.yaml` — active pipeline state (task_id, pipeline, step, step_status, gate_pending, context_budget, graph_available, history)
- `status.yaml` per task — persistent task metadata (description, developer, gates[], retries)
- `manifest.yaml` per task — checkpoint for resume (pipeline, developer, checkpoint)
- `telemetry.yaml` per task — written at completion

### 4c. Knowledge Flow
Three-tier access system (L0=index ~100-200 tokens, L1=summary ~500-2k tokens, L2=full ~2-10k tokens):
- 7 knowledge types: project-model, conventions, decisions, patterns, failures, quality-map, libraries
- Knowledge access matrix (canonical: `src/global/core/knowledge-access-matrix.yaml`) controls per-agent read access
- Write access restricted to: Mnemosyne (all types), Hermes (project_model only), Themis (quality_map only)

### 4d. Configuration Flow
- `config.yaml` — project configuration (version, stack, budgets, quality mode, audit thresholds, MCP, hooks, bootstrap state, graph)
- Read by orchestrator at pipeline start
- Schema defined in `src/schemas/config.schema.yaml` (32 fields across 12 sections)

## 6. API Contracts

### 6a. Agent Response Contract
Canonical source: `src/global/core/response-contract.yaml`
```
STATUS: success|failure|blocked|budget_exceeded
SUMMARY: <1-2 sentences, factual>
ARTIFACTS: [<list of file paths written>]
NEXT: <recommended next pipeline step>
```
Extended format for quality-gate agents adds:
```
QUALITY: {gate}={verdict} ({critical}C/{warning}W/{suggestion}S)
```

### 6b. Pipeline YAML Contract
Each pipeline YAML defines:
- `_meta` (name, description, trigger conditions)
- `steps[]` (id, agent, role, mode, writes_to, reads_from)
- `gates[]` (id, after_step, required, options[])
- `error_handlers` (per error type: action, max_attempts, display)
- `post` (reflection mode, budget_report)

### 6c. Rule Layer Assembly Contract
4-layer merge (L1 Base -> L2 Role -> L3 Project -> L4 Task):
- Inviolable rules (L1): never overridden
- Overridable rules (L1): can be overridden by L3 project rules
- Role constraints (L2): agent-specific NEVER rules
- Project rules (L3): stack, conventions, patterns, boundaries
- Task instructions (L4): assembled by Planner, never truncated

### 6d. Knowledge Access Contract
Canonical source: `src/global/core/knowledge-access-matrix.yaml`
- Per-agent read access levels (L0/L1/L2/null) for 7 knowledge types + graph
- Write access for 3 agents only (Mnemosyne, Hermes, Themis)
- Graph extras per agent (blast_radius, smells, metrics, etc.)

### 6e. Error Type Contract
11 error types (E1-E11) defined in pipeline YAMLs and `errors.md`:
- E1-INPUT, E2-SCOPE, E3-CONFLICT, E4-BUDGET, E5-QUALITY, E6-AGENT, E7-DRIFT, E8-STALE, E9-SEMANTIC, E10-DIVERGE, E11-TRUNCATION
- Each has defined action, max_attempts, escalation path

## 8. Key Architectural Invariants (from Constitution)

6 articles, 18 sub-articles defining inviolable properties:
- Art 1: Separation of concerns (orchestrator purity, agent single responsibility, no god components)
- Art 2: Determinism (pipeline determinism, gate determinism, no implicit decisions)
- Art 3: Transparency (traceability, budget visibility, error transparency)
- Art 4: Safety (no fabrication, user authority, rollback, escape hatch integrity)
- Art 5: Knowledge integrity (evidence-based, 3-confirmation threshold, consistency)
- Art 6: Self-protection (constitutional immutability, design doc authority, invariant verification)

Primary enforcement mechanism: `allowed-tools` in command frontmatter (structural). Secondary: `guard.sh` PostToolUse hook (detection + logging). Tertiary: CLAUDE.md prompt rules (behavioral).
