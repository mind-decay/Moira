# Modular Rules Architecture

## Four-Layer System

```
┌─────────────────────────────────────────────┐
│              AGENT RUNTIME RULES             │
│  (assembled per agent per task by Planner)   │
├─────────────────────────────────────────────┤
│ Layer 4: Task-specific instructions          │
│          "Modify these 3 files per plan"     │
├─────────────────────────────────────────────┤
│ Layer 3: Project rules                       │
│          "Use Zod for validation"            │
│          "Tailwind, not CSS modules"         │
├─────────────────────────────────────────────┤
│ Layer 2: Role rules                          │
│          "You are Implementer. You write     │
│           code. You don't make decisions."   │
├─────────────────────────────────────────────┤
│ Layer 1: Base rules                          │
│          "Never guess. Stop if uncertain."   │
│          "Write results to files."           │
│          "Return only status summary."       │
└─────────────────────────────────────────────┘
```

**Priority on conflict: Layer 4 > 3 > 2 > 1**

Exception: Layer 1 contains **inviolable rules** that CANNOT be overridden by any layer.

## Layer 1: Base Rules (core/rules/base.yaml)

Universal rules for ALL agents in ALL projects.

### Inviolable (cannot be overridden)

```yaml
inviolable:
  - "Never fabricate API endpoints, URLs, schemas, or data structures"
  - "Never proceed when information is insufficient — stop and report STATUS: blocked"
  - "Never suppress or ignore errors"
  - "Write all detailed output to state files, return only status summary"
  - "Never guess types, return formats, or data structures"
  - "Never commit secrets or credentials"
  - "Never modify files outside stated scope"
```

### Overridable defaults

```yaml
overridable:
  file_output_format: "markdown"
  max_function_length: 50
  naming_convention: "camelCase"
  test_framework: "jest"
  indent: "2 spaces"
```

## Layer 2: Role Rules (core/rules/roles/*.yaml)

Per-agent identity, capabilities, and constraints. See [agents.md](agents.md) for full role definitions.

Example — implementer.yaml:
```yaml
identity: |
  You are an Implementer. You write code according to the plan.
  You do NOT make architectural decisions.
  You do NOT deviate from the plan.
  If the plan is unclear — STOP and report STATUS: blocked.

capabilities:
  - Write code to project files
  - Read files specified in instructions
  - Use authorized MCP tools only

never:
  - "Never make decisions about WHAT to build (only HOW within plan)"
  - "Never add features not in the plan"
  - "Never refactor code outside of plan scope"
  - "Never add comments, docstrings, or type annotations to unchanged code"
```

## Layer 3: Project Rules (project/rules/*.yaml)

Generated at `/moira init`, refined over time.

### stack.yaml
```yaml
language: TypeScript
framework: Next.js 14
runtime: Node.js 20
styling: Tailwind CSS
orm: Prisma
testing: Jest + React Testing Library
ci: GitHub Actions
```

### conventions.yaml
```yaml
naming:
  files: kebab-case
  components: PascalCase
  functions: camelCase
  constants: UPPER_SNAKE_CASE
  types: PascalCase

formatting:
  indent: 2 spaces
  quotes: single
  semicolons: true
  max_line_length: 100

structure:
  # Note: `structure:` section is populated from the structure scanner's `dir_*`
  # frontmatter fields (e.g., dir_source, dir_tests, dir_config), not from presets.
  # See D-060 and design/specs/archive/2026-03-13-bootstrap-scanner-reform.md.
  components: src/components/{feature}/{ComponentName}.tsx
  pages: src/app/{route}/page.tsx
  api: src/app/api/{resource}/route.ts
  services: src/services/{name}.ts
  types: src/types/{name}.ts
  schemas: src/schemas/{name}.ts
  utils: src/utils/{name}.ts
  tests: __tests__/{name}.test.ts (co-located)
```

### patterns.yaml
```yaml
data_fetching: "Server components + server actions"
client_state: "Zustand stores in src/stores/"
server_state: "React Query with custom hooks in src/hooks/"
validation: "Zod schemas, validate at API boundary"
error_handling: "Error middleware + typed error classes"
db_access: "Repository pattern, src/repositories/"
auth: "JWT in httpOnly cookies, middleware-based"
```

### boundaries.yaml
```yaml
do_not_modify:
  - "src/legacy/**"           # Scheduled for removal
  - "prisma/migrations/**"    # Auto-generated
  - "node_modules/**"
  - ".next/**"

modify_with_caution:
  - "prisma/schema.prisma"    # Requires migration plan
  - "src/middleware.ts"        # Affects all routes
  - "next.config.js"          # Build configuration
```

## Layer 4: Task-Specific Instructions

Generated by Planner for each agent invocation in a specific task.

Contains:
- Specific files to read/modify
- Contract interfaces (for parallel batches)
- MCP tool authorization
- Context from previous pipeline steps
- Budget limits

See [pipelines.md](pipelines.md) for instruction assembly examples.

## Rule Assembly Process

Planner assembles rules as part of planning step:

1. Load base.yaml (Layer 1)
2. Load role file for target agent (Layer 2)
3. Load relevant project rules (Layer 3)
4. Add task-specific instructions (Layer 4)
5. Check for conflicts between layers
6. Resolve conflicts (higher layer wins, except inviolable)
7. Write assembled instructions to: `state/tasks/{id}/instructions/{agent}.md`

### Conflict Detection

```markdown
## Conflict Report (in plan.md)

Layer 2 (implementer.yaml): "Write functions ≤50 lines"
Layer 3 (conventions.yaml): "Reducer functions can be up to 100 lines"

Resolution: Layer 3 wins (project-specific override).
Applied rule: "Functions ≤50 lines, except reducers ≤100 lines"
```

### Simplified Assembly (Pre-Planning Agents)

For agents dispatched before the Planner runs (Classifier, Explorer, Analyst) and for Quick Pipeline agents, the orchestrator assembles rules directly:

1. **Layer 1 (Base):** `core/rules/base.yaml` — universal rules
2. **Layer 2 (Role):** `core/rules/roles/{agent}.yaml` — agent-specific rules
3. **Layer 3 (Project):** Project rules from `config.yaml` → `rules` section
4. **Layer 4 (Task):** Task-specific instructions from orchestrator context (task description, classification result, scoping)

This is a lightweight version of the full assembly. No conflict detection (Layers 1-2 are pre-validated, Layer 3 is user-configured). Knowledge is loaded per the access matrix at the appropriate level.

See `src/global/skills/dispatch.md` Simplified Assembly section for implementation. Per D-041.

## Rule Updates

Rules are updated through two paths:

### Path 1: Reflection → Proposal → Approval
- Reflector observes issue across 3+ tasks
- Proposes rule change with evidence
- User approves → rule updated

### Path 2: Audit → Recommendation → Approval
- Auditor finds inconsistency between rules and reality
- Recommends update with evidence
- Batch approval (low-risk auto, high-risk manual)

### Path 3: User Direct
- User manually edits rules in .moira/
- Next task picks up changes automatically
- Audit verifies consistency on next run
