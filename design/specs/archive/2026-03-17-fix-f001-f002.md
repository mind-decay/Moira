## Fix 1: Lightweight Reflection as Explicit Completion Step (F-001)

### Problem

Orchestrator.md Section 7 completion flow has 18+ numbered action items under the `done` action (lines 363-385), followed by Xref Consistency Check, then a Reflection Dispatch reference table (lines 407-420). The table is not a numbered step — it's a dangling reference section that the orchestrator may skip because the pipeline already appears "complete" after step 385 sets status to `completed`.

The table entry for `lightweight` starts with "No Reflector dispatched" which reads as "do nothing", though the second sentence says to write a minimal note.

### Fix

Convert the Reflection Dispatch section into an explicit numbered action item within the `done` completion actions, positioned **before** setting pipeline status to `completed`.

Specifically:
1. Move reflection dispatch from a separate reference section into the `done` action list, before the `Set pipeline status to completed` step
2. Inline the lightweight case directly: "Read pipeline's `post.reflection` value. If `lightweight`: write reflection note to `state/tasks/{id}/reflection.md` using `templates/reflection/lightweight.md` (substitute placeholders). Otherwise: invoke `reflection.md` skill for agent dispatch."
3. Remove the standalone Reflection Dispatch table (lines 407-420) — its content is now part of the completion step

**Note on ordering:** Moving reflection before `Set pipeline status to completed` means that for `deep` and `epic` modes (foreground/blocking Mnemosyne dispatch), the pipeline remains open during reflection. This is intentional — reflection is part of the task lifecycle and should complete before the task is marked done. For `background` mode, Mnemosyne is dispatched non-blocking, so the pipeline proceeds to completion without waiting.

### Files to Modify

|------|--------|
| `src/global/skills/orchestrator.md` | Add reflection as explicit step in `done` actions; remove standalone table |

### Success Criteria

- Reflection dispatch is a numbered step in the `done` completion action list
- The step appears before `Set pipeline status to completed`
- Lightweight case is clear and actionable (no "No X dispatched" negative phrasing)
- Reference to `reflection.md` skill preserved for non-lightweight modes

## Fix 2: Post-Agent Guard Check (F-002)

### Problem

PostToolUse hooks (`guard.sh`, `budget-track.sh`) fire only in the main orchestrator session. Agents dispatched via the Agent tool run as separate subprocesses that don't inherit hooks. This means:
- Guard Layer 2 is bypassed for all agent work (where ALL project file modifications happen)
- Budget tool-call tracking has no data from agents

Key insight: guard.sh hook cannot block anyway (D-075, PostToolUse fires after the fact). It only detects and logs. A post-agent git diff check is strictly stronger — it can actually **block the pipeline**.

### Architecture Decision

**Post-agent git diff verification** replaces hook-based detection for agents. This is not a workaround — it's an upgrade:
- Hooks detect after each tool call but cannot block → post-agent diff detects after agent completion and CAN block
- One git diff call per agent vs dozens of hook invocations → more efficient
- No dependency on Claude Code subprocess architecture → more robust

Guard.sh continues to serve its original purpose for orchestrator-level detection (Layer 2 for orchestrator). Post-agent diff becomes Layer 2 for agents.

Budget tracking: accept agent self-reporting via `moira_state_agent_done` as authoritative. Hook-based tracking only logs tool_name + file_path + file_size — it doesn't provide token counts. Self-reporting already provides the budget-relevant data.

### Protected Paths

Agents MUST NOT modify:
- `design/CONSTITUTION.md` — absolute prohibition (Art 6.1)
- `design/**` — design docs without approval (Art 6.2)
- `.claude/moira/config/**` — system configuration
- `.claude/moira/core/**` — core rules and pipelines
- `src/global/**` — Moira source code (agents, skills, hooks, templates, lib)

Agents ARE allowed to modify:
- `.claude/moira/state/tasks/{current_task_id}/**` — task artifacts
- `.claude/moira/knowledge/**` — knowledge base entries (deep scan, quality map updates)
- `.claude/moira/state/current.yaml` — pipeline state (only via shell lib functions, but structurally allowed)
- `.claude/moira/state/queue.yaml` — epic queue (decomposition pipeline)
- Project source files — that's their job

### Where in Pipeline

**After each file-modifying agent returns.** Specifically, after the orchestrator parses the agent response (Section 2, step d) and before advancing to the next step (step g):

- After Hephaestus (implementer) — always modifies files
- After Hermes (explorer) in write-mode — when creating context.md/exploration.md artifacts that touch project analysis
- After any agent that returns `ARTIFACTS` referencing modified project files

NOT after read-only agents (Apollo/classifier, Athena/analyst) or review agents (Themis — reads but doesn't write project files).

### Violation Response

When protected file modification detected:

1. **Block pipeline** — do not advance to next step
2. **Display violation** — show which protected files were modified and by which agent
3. **Present options:**
   - `revert` — revert the protected file changes (`git checkout -- <files>`), keep other changes, continue pipeline
   - `accept` — accept the changes (user explicitly overrides), continue pipeline
   - `abort` — abort the pipeline

Log the violation to `state/violations.log` regardless of user choice.

### Guard Check Mechanism

```
1. Get pre-agent HEAD (or use status.yaml git.pre_task_head for first agent)
2. Run: git diff --name-only HEAD (unstaged) + git diff --name-only --cached (staged)
3. Filter modified files against protected paths list
4. If any protected file modified → violation flow
5. If clean → proceed
```

Protected paths list is defined in the orchestrator skill directly (not in a separate config file — keeps it simple, one source of truth alongside the check logic).

### Post-Agent Guard vs E7-DRIFT

The existing E7-DRIFT error code (in `errors.md`) describes orchestrator violations detected by guard.sh — non-blocking, log-only. The post-agent guard check is a different mechanism: it's a **guard gate** (like the Xref Warning Gate), not an error code. It blocks the pipeline with user choice.

These are separate flows:
- **E7-DRIFT** — orchestrator hook detection, non-blocking, log + context warning. Stays as-is.
- **Guard Violation Gate** — post-agent git diff detection, blocking, revert/accept/abort. New mechanism.

E7-DRIFT's description in `errors.md` should note that it covers orchestrator-level violations only, and reference the Guard Violation Gate for agent-level violations.
